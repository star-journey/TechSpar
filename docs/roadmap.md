# 开发计划

这页记录当前准备推进的功能计划，方便后续按优先级持续补充和拆分。

## 计划一：训练题目多端同步

目标：当用户开始一次训练答题后，无论是关闭页面重新打开，还是切换到其他电脑、手机等设备，都能继续未完成的答题过程。

### 拆解

- 训练会话需要有可恢复的唯一标识。
- 答题进度需要定期持久化到后端。
- 重新进入训练时，前端能够根据会话状态恢复题目、已答内容和当前进度。
- 不同设备登录同一账号后，应能读取同一份未完成训练状态。

### 里程碑

1. 定义训练会话模型，明确哪些状态需要落库，哪些状态可以在前端短暂缓存。
2. 在答题过程中增加自动保存机制，保证页面刷新或误关页面后可恢复。
3. 在训练入口增加“继续上次训练”的恢复流程。
4. 统一多设备读取逻辑，确保同账号下的未完成训练状态一致。

### 验收标准

- 同一账号在不同设备打开后，能恢复到同一个未完成训练会话。
- 页面刷新、标签页关闭、浏览器重启后，已答内容不会丢失。
- 恢复后的题目顺序、当前题号和已提交答案保持一致。
- 完成训练后，旧会话会正确结束，不再继续被误恢复。

### 风险

- 训练过程中频繁写库可能带来性能和一致性问题。
- 多设备同时操作时，需要明确“以哪个会话状态为准”。
- 如果题目生成依赖实时上下文，恢复逻辑要避免重复出题或题目错位。

## 计划二：增强项目出题时的 RAG 能力

目标：在项目相关出题时，提升检索增强生成的准确性、上下文覆盖和问题贴合度，让题目更贴近项目内容、技术栈和真实面试追问路径。

### 拆解

- 优化项目资料的切分、索引和召回策略。
- 将项目描述、历史问答、知识库和画像信息更好地融合到出题上下文中。
- 针对项目类型、技术栈和历史薄弱点做更精准的检索加权。
- 增加对生成题目与项目材料一致性的校验。

### 里程碑

1. 梳理项目出题所依赖的数据源，区分项目简介、简历内容、知识库和历史训练记录。
2. 改进检索策略，让项目材料的召回更完整，减少只命中表层关键词的问题。
3. 设计面向项目出题的上下文拼装策略，把项目事实、技术栈和用户画像组合进提示词。
4. 增加生成后校验，过滤与项目内容不一致、过于泛化或重复度过高的题目。
5. 补充评测集，持续观察不同项目场景下的命中率和题目质量。

### 验收标准

- 项目出题时，问题能明显引用项目本身的技术点、场景或实现细节。
- 题目与项目材料不一致、过于空泛的问题数量显著下降。
- 同一个项目在不同轮次出题时，能基于历史问答做合理去重和递进。
- 能区分不同项目类型，避免检索策略对某一类项目过度偏置。

### 风险

- 项目材料质量差或过短时，RAG 提升空间有限。
- 上下文拼装过多会带来提示词膨胀，影响稳定性和成本。
- 评测标准不清时，容易只看到“更长的答案”，却不一定真的更准。

- 若计划进入实施阶段，再补充里程碑、依赖和验收标准。
